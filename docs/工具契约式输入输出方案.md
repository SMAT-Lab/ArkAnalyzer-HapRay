# 工具契约式输入输出方案

本文档描述 `HapRay ` **perf_testing** 与 **tools** 三件套（optimization_detector、static_analyzer、symbol_recovery）在 Agent / 自动化场景中的 **统一运行结果契约（tool-result）**：设计原则、系统级约定及各工具落地方式。

---

## 1. 背景与目标

### 1.1 问题

- 编排脚本与 Agent 需要 **稳定、可解析** 的方式判断：命令是否成功、退出码、产物路径、错误原因。
- 仅依赖 **人类可读日志** 或 **单一业务报告路径（如 `-o`）** 不足以表达「本次 CLI 运行」的元信息；多格式输出时路径规则也更复杂。

### 1.2 目标

- 约定 **统一 JSON 结构（HapRay tool-result v1）**，与具体业务报告（Excel、静态分析目录等）分离。
- **优先落盘**：在可确定输出目录或约定路径时，写入 **`hapray-tool-result.json`**（或与 `--result-file` 显式路径一致）。
- **stdout 兜底**：仅在 **无法写文件** 或 **无可用契约路径** 时，通过 **`--machine-json`** 输出一行 JSON（日志走 stderr，避免污染管道）。

### 1.3 业务报告 vs 契约

| 维度 | 业务报告（如 `-o` 指向的文件） | tool-result 契约 |
|------|-------------------------------|------------------|
| 内容 | 领域数据（优化检测结果、静态分析结果等） | 本次运行的成功/失败、exit_code、版本、错误、**产物路径索引** |
| 读者 | 开发/分析人员、下游分析 | Agent、CI、编排器 |
| 是否必需人工阅读 | 是 | 否；自动化场景才依赖 |

## 2. 统一数据模型（tool-result v1）

### 2.1 Schema

- 规范文件：`docs/schemas/hapray-tool-result-v1.json`（JSON Schema draft 2020-12）。
- 运行时 **建议** 使用 `schema_version: "1.0"`，便于后续演进。

### 2.2 核心字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `schema_version` | string | 固定 `"1.0"` |
| `tool_id` | string | `perf_testing` \| `optimization_detector` \| `static_analyzer` \| `symbol_recovery` |
| `tool_version` | string | 可选，工具版本 |
| `action` | string | 可选，子命令/动作名 |
| `success` | bool \| null | 是否成功；部分历史 action 返回 `None` 时可为 `null` |
| `exit_code` | int | 与进程退出码语义对齐 |
| `outputs` | object | 自由扩展；常含 `hapray_tool_result_path`、各工具特有字段 |
| `error` | string \| null | 失败时的短码或信息 |

### 2.3 通用 outputs 约定

- **`hapray_tool_result_path`**：本契约 JSON 文件的绝对路径；**在写入磁盘前**即写入 `outputs`，保证文件内容与内存一致。
- **`hapray_tool_result_sink`**：若为 `"stdout"`，表示本次契约通过 **stdout 一行 JSON** 输出（兜底）。
- 各工具可在 `outputs` 中增加 **`report_files`**、**`reports_by_format`** 等，见下文分工具说明。

### 2.4 实现入口（概念）

- Python：`finalize_contract(payload, result_path=..., machine_json=...)`，内部先设置 `outputs.hapray_tool_result_path`，再 `write_tool_result_file`。
- 各包内模块名：`machine_output.py`（或 `hapray/core/common/machine_output.py`）。

---

## 3. 系统级约定

### 3.1 输出优先级

1. **有 `result_path`**：写入该路径（父目录不存在则创建）。
2. **写文件失败** 且提供了 **`--machine-json`**：同一 `payload` 打一行到 stdout，并标注 `hapray_tool_result_sink`。
3. **无 `result_path`** 且 **`--machine-json`**：仅 stdout 一行。
4. **无路径且未带 `--machine-json`**：不写契约文件、不打 stdout（静默跳过契约；业务侧仍可能有日志与业务报告）。

### 3.2 默认契约文件名

- 固定 basename：**`hapray-tool-result.json`**（除非工具另有 `--result-file` 覆盖，见分工具说明）。

### 3.3 `--machine-json` 默认值

- **默认关闭**（`argparse` 使用 `action='store_true'` 或未传参等价 `False`）。
- 仅在需要 **管道消费** 或 **无写盘路径** 时显式开启。

### 3.4 日志与 stdout

- 当 **`--machine-json` 生效且契约走 stdout** 时，控制台日志应走 **stderr**，避免与 JSON 行混淆（各工具在 `machine_json=True` 时对 `StreamHandler` 或 log4js 做 stderr 配置）。

## 4. 各工具详细整改方案

### 4.1 `perf_testing`（`scripts/main.py`）

| 项目 | 说明 |
|------|------|
| **入口** | `python scripts/main.py [全局选项] <action> [子参数]` |
| **全局剥离参数** | `--machine-json`；`--result-file <path>`（写入契约的绝对/相对路径解析为绝对路径） |
| **契约路径解析顺序** | ① `--result-file`；② `execute` 返回 `(exit_code, reports_path)` 且 `reports_path` 非空时按 action 落盘（目录或 Excel 旁）；③ 否则对 `static` / `compare` / `update` 从子参数解析（与现有 macOS 路径映射逻辑对齐） |
| **Action.execute 返回值** | 强制为 **`tuple[int, str]`**：`(exit_code, reports_path)`，`exit_code==0` 表示成功；`reports_path` 为报告根目录或主产物路径（失败或无路径时可为 `''`）。定义见 `hapray/core/common/action_return.py`。非法类型在 `main.py` 中打告警。 |
| **实现模块** | `hapray/core/common/machine_output.py`：`build_tool_result`、`finalize_contract`、`finish_perf_testing_contract`、`interpret_perf_testing_return`；`hapray/core/common/action_return.py`：`ActionExecuteReturn`（= `tuple[int, str]`）、`is_valid_action_execute_return` |
| **日志** | `configure_logging(..., machine_json=...)`：为 true 时控制台走 stderr |
| **注意** | 子进程 `--multiprocessing-fork` 不发射契约；`execute` 返回 `(exit_code, reports_path)` 时 `success` 与 `exit_code` 由解释器唯一确定（均为布尔/整数，非 `null`）。Schema 仍允许 `success: null` 以兼容其它工具或历史载荷。 |
| **gui-agent 扩展 outputs** | 当 `reports_path` 非空时，在 `outputs` 中合并 **`gui_agent`** 与 **`partial`**（见 §5.2），由 `enrich_gui_agent_contract_outputs` 扫描报告目录；**不含**思考全文，仅 `message_excerpt` / `action_excerpt` 等短字段。 |

**无 `--result-file` 时契约落盘路径（节选）**：各 action 的 `reports_path` 为 `execute` 返回元组第二段；非空时：`perf` / `prepare` / `haptest` / `gui-agent` / `ui` / `ui-compare` / `update`（目录）→ `<abs(reports_path)>/hapray-tool-result.json`；`hilog` / `compare`（Excel 文件路径）→ `<dirname(abs)>/hapray-tool-result.json`。

#### 4.1.1 全局约定（与 `main.py` / `machine_output` 对照）

本文节与 `perf_testing/scripts/main.py` 中注册的 action、`hapray.core.common.action_return.ActionExecuteReturn` 及 `hapray.core.common.machine_output.resolve_perf_testing_result_path` 的实现**对照**，供 Agent / CI 与维护者查阅。

| 项 | 约定 |
|----|------|
| `execute(sub_args)` 返回值 | **`tuple[int, str]`**：`(exit_code, reports_path)`；`exit_code == 0` 为成功 |
| 非法返回 | `main.py` 会打告警：`Action %s 的 execute 返回了非契约类型` |
| 进程退出码 | `main.py` 末尾 **`sys.exit(ret[0])`**，与 `tool-result` 的 `exit_code` 一致 |
| 契约发射 | 子进程 `--multiprocessing-fork` **不**写契约、不 stdout |
| 契约落盘优先级 | `--result-file` → 由 `reports_path` / 子参数解析 → `--machine-json`（与 §3.1、上表「契约路径解析顺序」一致） |

#### 4.1.2 各 Action 对照表

| action | 实现文件 | `execute` 均返回 `(int, str)` | 成功时 `reports_path` 典型含义 | 默认 `hapray-tool-result.json` 路径（无 `--result-file`） |
|--------|----------|------------------------------|--------------------------------|--------------------------------------------------------------|
| `perf` | `perf_action.py` | 是 | 报告根目录（字符串路径） | `<abs(reports_path)>/hapray-tool-result.json` |
| `static` | `static_action.py` | 是 | 成功时当前为 **`''`**（空） | 由 **`_parse_static_result_path(sub_args)`** 根据 `-o` 等解析（macOS 经用户目录映射），不依赖 `reports_path` |
| `update` | `update_action.py` | 是 | 更新后的 **`report_dir`** | `<abs(report_dir)>/hapray-tool-result.json` |
| `compare` | `compare_action.py` | 是 | 生成的 **Excel 文件路径** | `<dirname(abs(excel))>/hapray-tool-result.json`；失败时亦可经 `_parse_compare_result_path` 兜底 |
| `prepare` | `prepare_action.py` | 是 | 每次执行在 **`~/ArkAnalyzer-HapRay/prepare/`（macOS）或 `./prepare/`（其他平台）** 下创建的 **会话子目录**（时间戳命名） | `<abs(session_dir)>/hapray-tool-result.json` |
| `ui` | `ui_action.py` | 是 | 分析输出目录 | `<abs(reports_path)>/hapray-tool-result.json` |
| `ui-compare` | `ui_compare_action.py` | 是 | `--output` 目录（绝对路径） | `<abs(output)>/hapray-tool-result.json` |
| `hilog` | `hilog_action.py` | 是 | 主产物文件路径 | `<dirname(abs(file))>/hapray-tool-result.json` |
| `gui-agent` | `gui_agent_action.py` | 是 | 报告根目录 | `<abs(reports_path)>/hapray-tool-result.json`；fork 子进程返回 `(0,'')` 且不发射契约 |
| `haptest` | `haptest_action.py` | 是 | 报告根目录 | `<abs(reports_path)>/hapray-tool-result.json` |

#### 4.1.3 后续可核对项

| 优先级 | 项 | 说明 |
|--------|----|------|
| 核对 | **`static`** | 成功时 `reports_path` 仍为空属当前设计；失败路径 CLI 冒烟见 `tests/test_contract_cli_smoke.py` 与 §5.3。 |

## 5. 与 CI / Agent 集成建议

1. **首选读文件**：轮询或步骤结束后读取 **`hapray_tool_result_path`** 指向的文件（或直接读约定路径）。
2. **解析**：单行 JSON（stdout）与多行 pretty-print（文件）需统一用 JSON parser；文件为 **UTF-8**。
3. **失败策略**：同时检查 `exit_code` 与 `success`；`outputs.error` 为人类/短码可读信息。
4. **多仓库路径**：macOS App / 映射目录下，业务路径与契约路径可能在 `~/ArkAnalyzer-HapRay/...`，与开发机 cwd 不同，**以契约内绝对路径为准**。

### 5.1 Tauri 桌面端（`plugin.json`）

- 仓库内 **`perf_testing/plugin.json`**、**`tools/optimization_detector/plugin.json`**、**`tools/static_analyzer/plugin.json`**、**`tools/symbol_recovery/plugin.json`** 均已配置顶层 **`tool_contract`**，与各自 CLI 的 `--result-file` / `--machine-json` 行为一致。
- 插件可在顶层声明 **`tool_contract`**，与 CLI 全局参数对齐，避免「界面一套、命令行一套」：
  - **`enabled`**：为 **true** 时，在启动子进程前注入 **`--result-file`**，路径为 **`~/.hapray-gui/results/<plugin_id>/<timestamp>/<result_file>`**（默认文件名为 **`hapray-tool-result.json`**，可由 **`result_file`** 覆盖）。
  - **`machine_json`**：为 **true** 时额外注入 **`--machine-json`**（与 CLI 一致；默认 **false**，GUI 以**契约落盘**为主，便于同时采集子进程 stdout 日志）。
- 子进程结束后**优先**解析上述契约文件，读取 **`outputs.reports_path`** 作为「打开报告目录」等后续操作的依据；若缺失或解析失败，再回退到日志启发式与表单参数（与实现见 `desktop/src-tauri/src/commands.rs`）。

### 5.2 GUI Agent / LLM 环路可观测性（`schema_version` 1.0）

- **一次性契约（当前默认）**：`gui-agent` 子进程**正常退出**时写入 **`hapray-tool-result.json`**，此时约定 **`outputs.partial: false`**，表示本文件为**本轮跑完后的完整快照**；结构化步骤边界在 **`outputs.gui_agent`** 内，供外层 LLM **续跑/改参**时直接读 JSON，而不依赖日志解析。
- **`outputs.gui_agent` 要点**：
  - **`contract_mode`**：当前为 **`"final"`**（与 `partial: false` 一致）。
  - **`reports_dir`**：报告根目录绝对路径。
  - **`scenes[]`**：按 `应用包名/场景目录` 扫描；每项除 **`step_count`**、**`perf_steps_summary`**（来自 `steps.json` 的 `stepIdx`/`name`/`description`）外，含 **`llm_step_events`**：从 **`ui/stepN/pages.json`** 聚合的 LLM 步列表，每步含 **`perf_step_id`**、**`page_idx`**、**`step_index`**、**`finished`**、**`success`**、**`error_code`**（失败短码，成功为 `null`）、**`message_excerpt`** / **`action_excerpt`**；另含 **`last_llm_finished`**、**`last_llm_error_code`** 便于快速判断末步状态。
- **附属落盘**：每步详细字段同时写入场景目录下 **`ui/stepN/pages.json`** 内 `gui_agent` 对象（`gui_agent_runner` 写入 **`step_index`**、**`finished`**、**`success`**、**`error_code`** 等）；契约层为摘要，全文仍在报告文件中。
- **增量契约（预留）**：若将来支持**长时间运行**中多次覆盖 **`hapray-tool-result.json`**，约定 **`outputs.partial: true`** 表示**尚未结束**，仅反映当前进度；**`schema_version` 仍为 `1.0` 时优先只扩展 `outputs`**，不升 Schema 主版本。

**与实现细节对齐（`gui-agent` 写入前合并）**：当 **`action == gui-agent`** 且 **`outputs.reports_path` 非空** 时，在写入 `hapray-tool-result.json` 前合并 **`outputs.gui_agent`** 与 **`outputs.partial`**：

- **`partial`**：当前实现恒为 **`false`**，表示进程结束时**一次性**完整契约；若将来支持长跑中多次刷盘，约定为 **`true`**（与上文「增量契约」一致）。
- **`reports_dir`**：报告根目录绝对路径。
- **`contract_mode`**：当前为 **`"final"`**（与 `partial: false` 一致）。
- **`scenes`**：按 `应用包名/场景子目录` 扫描，每项含 **`app_package`**、**`scene_dir`**、**`test_info_present`**、**`steps_present`**、**`step_count`**、**`perf_steps_summary`**、**`llm_step_events`**、**`last_llm_finished`**、**`last_llm_error_code`** 等；`llm_step_events` 由 **`ui/stepN/pages.json`** 中 `gui_agent` 聚合（含 **`step_index`**、**`finished`**、**`error_code`** 等）。
- 不包含 LLM 思考全文；详细字段在报告目录的 `pages.json` 中。

### 5.3 CI 契约冒烟

- **Workflow**：`.github/workflows/build.yml` 中的 **`contract-smoke-*` job**（在 **`build` 矩阵全部成功** 之后运行，与主构建同一次触发）。
- **内容**：`perf_testing` 的 Schema 单测 + `static` 失败路径 CLI 冒烟；`tools/optimization_detector`（空目录输入）；`tools/symbol_recovery`（缺失 `perf.data`）。
- **说明**：`main.py` 在 `execute` 后 **`sys.exit(exit_code)`**，与 `tool-result` 中 `exit_code` 一致；若进程恒为 0 退出则 CI/Agent 易误判。

**`static_analyzer`（HAP）**：未纳入上述 workflow；构建链依赖 **Bun** 等（见 `tools/static_analyzer/package.json`），与轻量 Python 冒烟分离；契约实现见 §4.3。

---

## 6. 演进与兼容性

- **`schema_version`**：新字段尽量 **附加** 在 `outputs` 或顶层可选字段，避免破坏现有解析。
- **工具版本**：建议在契约中带 `tool_version`，便于统计与排障。
